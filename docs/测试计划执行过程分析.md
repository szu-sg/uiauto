# 测试计划执行过程分析

基于当前 uiauto 仓库代码，对「测试计划」从创建到执行完成的全流程做梳理。

---

## 一、整体数据流概览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ 触发方式                                                                     │
│ • 手动：前端 PlanDetail 点「执行」→ POST /api/runs { plan_id }               │
│ • 定时：scheduler 按 cron 触发 → 插 runs → executePlan(runId, plan)          │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 后端 runs 路由 (backend/src/routes/runs.js)                                 │
│ POST / → 校验 plan_id → INSERT runs (status='pending') → executePlan()     │
│ 返回 201 { id: runId, plan_id, status: 'pending' }                           │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ 执行器队列 (backend/src/services/executor.js)                               │
│ executePlan(runId, plan) 入 p-queue（全局 concurrency: 1，同一时间只跑一个） │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                    队列消费时按顺序执行以下阶段
                                        │
    ┌───────────────────┬───────────────────┬───────────────────────────────┐
    │ 1. 写 run_cases   │ 2. 克隆/拉取仓库   │ 3. 安装依赖                   │
    │ 先插入所有用例行   │ ensureRepo()      │ installRepoDeps()             │
    │ (pending)，便于    │ progress_phase=   │ progress_phase='installing'   │
    │ 取消时报告也有列表  │ 'cloning'         │ 复制 .env，注入 auth.json     │
    └───────────────────┴───────────────────┴───────────────────────────────┘
                                        │
                                        ▼
    ┌───────────────────────────────────────────────────────────────────────┐
    │ 4. 执行用例 (progress_phase='running')                                 │
    │ 内层 PQueue concurrency=RUN_CONCURRENCY（默认 3），多用例/多浏览器并行  │
    │ 每个任务：runPlaywrightTest() → 拷贝产物 → 更新 run_cases 状态          │
    └───────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
    ┌───────────────────────────────────────────────────────────────────────┐
    │ 5. 收尾                                                                 │
    │ 写 run.log，更新 runs.status='done'（或 cancelled/failed）             │
    │ result_dir、log_text 写入 DB；runningTask 清空                          │
    └───────────────────────────────────────────────────────────────────────┘
```

## 二、核心组件与职责

### 2.1 触发入口

| 来源 | 文件 | 行为 |
|------|------|------|
| **手动执行** | `frontend/src/pages/PlanDetail.jsx` | `startRun()` → `POST /api/runs` body: `{ plan_id }` → 跳转 `/runs/:id` |
| **定时执行** | `backend/src/services/scheduler.js` | `node-cron` 按 plan 的 `schedule_cron` 触发 → `triggerRun(planId)` → 插 `runs` → `executePlan()` |
| **API** | `backend/src/routes/runs.js` | `POST /` 校验 `plan_id`，查 plan，插 run，调 `executePlan(runId, plan)`，返回 `{ id, plan_id, status }` |

### 2.2 执行器 (executor.js)

| 概念 | 说明 |
|------|------|
| **全局队列** | `p-queue` 且 `concurrency: 1`，保证同一时刻只执行一个计划（一个 run）。 |
| **run 状态** | `pending` → 队列接手后立刻 `running`；结束为 `done` / `failed` / `cancelled`。 |
| **进度阶段** | `progress_phase`: `cloning` → `installing` → `running`，供 `/api/runs/status` 查询。 |
| **任务展开** | 由 `plan.cases_json` 与 `plan.run_browsers_json` 展开为 (casePath, browser) 列表；若无 run_browsers 则 browser=null（默认 project）。 |
| **单用例执行** | `runPlaywrightTest(repoDir, casePath, runId, caseRowId, taskRef, browser)`：在仓库目录 `npx playwright test <casePath>`，可选 `--project <browser>`；chrome/msedge 会映射为 chromium。 |
| **并发** | 用例级用内层 `PQueue`，`RUN_CONCURRENCY`（默认 3）个 Playwright 进程同时跑。 |
| **取消** | `cancelRun(runId)` 对应当前 `runningTask.runId` 时 kill 子进程；执行循环内每次跑用例前查 run 是否已 `cancelled`。 |

### 2.3 仓库与环境

| 步骤 | 实现 |
|------|------|
| **仓库目录** | `backend/repos/<owner>_<repo>`，已有则 `fetch` + `checkout` + `reset --hard origin/branch`，否则 `clone --branch`。 |
| **超时** | `CLONE_TIMEOUT_MS` / `PULL_TIMEOUT_MS`（可环境变量覆盖），超时抛错 → run 标记为 failed。 |
| **依赖** | 仓库内 `npm install` + `npx playwright install --with-deps`。 |
| **登录态** | 从 `data/run.env` 或 `data/.env` 等读 `WPS_SID`，在仓库内生成 `auth.json`（或手写 JSON），供 Playwright `storageState` 使用。 |

### 2.4 结果落盘与 DB

| 项 | 说明 |
|------|------|
| **run_cases** | 先插入本 run 所有 (case_path, browser) 为 `pending`；每用例跑完后更新：status、duration_ms、error_message、screenshot_path、video_path、trace_path、log_path。 |
| **产物目录** | 仓库内 `test-results-<caseRowId>` 拷贝到 `backend/results/<runId>/<caseId>/`，路径相对 results 写回 DB。 |
| **日志** | 每用例 stdout/stderr 写入 `results/<runId>/<caseId>/log.txt`；整 run 汇总写入 `results/<runId>/run.log`，并截断后写入 `runs.log_text`。 |

### 2.5 取消与状态查询

| 接口 | 行为 |
|------|------|
| `POST /api/runs/:id/cancel` | 仅当 status 为 pending/running 时，将 run 置为 `cancelled`，并调 `cancelRun(id)` kill 当前 run 的子进程。 |
| `GET /api/runs/status` | 返回当前是否在执行、runId、progress_phase（及中文 phaseLabel）。 |

## 三、数据库与前端对应关系

### 3.1 主要表

- **plans**：计划元信息；`cases_json` 用例路径列表，`run_browsers_json` 可选浏览器列表，`schedule_cron` / `schedule_enabled` 定时。
- **runs**：某次执行；`plan_id`、`status`、`started_at`、`finished_at`、`result_dir`、`log_text`、`progress_phase`。
- **run_cases**：某次 run 下每条用例（含 browser）的执行结果与产物路径。

### 3.2 前端执行入口

- **PlanDetail**：展示计划详情与「执行」按钮；点击后 `POST /api/runs`，成功后跳转 ` /runs/<runId>`（Run 详情页）。

---

## 四、执行过程小结（按时间顺序）

1. **触发**：用户或定时任务创建一次 run（`runs` 表插入，status= pending）。
2. **入队**：`executePlan(runId, plan)` 加入全局队列；队列空闲时开始执行该 run。
3. **准备**：  
   - 写入本 run 所有 `run_cases`（pending）。  
   - 更新 run 为 running，progress_phase= cloning → installing；  
   - 克隆/拉取仓库、安装依赖、复制 .env、注入 auth.json。
4. **执行**：progress_phase= running；按 (casePath, browser) 任务列表，经内层 PQueue 并发调用 `runPlaywrightTest`；每完成一条则拷贝产物、写 log、更新对应 run_case。
5. **结束**：所有任务完成后写 run.log，更新 runs 为 done（若中途被取消则 cancelled；异常则 failed），清空 runningTask。
6. **取消**：若用户在此期间调用 cancel，DB 先置 run 为 cancelled 并 `cancelRun`；执行循环在下一个用例前检查到 cancelled 即不再跑新用例并按结束逻辑收尾。

以上即为当前仓库中「测试计划」从触发到执行完成（或取消/失败）的完整过程；与 `docs/executor-evaluation.md` 中的设计一致，并补充了与前端、调度器、runs 路由的衔接细节。
