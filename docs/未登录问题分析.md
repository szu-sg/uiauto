# 未登录问题分析：Cookie 未注入 vs 操作过快

## 一、结论（直接回答）

**根因是 Cookie 未注入或注入的 Cookie 无效**，而不是「操作太快、注入的 Cookie 还没生效」。

理由简述：

- Playwright 的 `storageState`（auth.json）是在**创建浏览器上下文时**就加载的，第一个请求（例如 `page.goto('https://365.kdocs.cn/...')`）**已经会带上这些 Cookie**，不存在「先发请求再等 Cookie 生效」的时序问题。
- 你提供的截图里，当前页面是 **account.wps.cn 登录页**，说明 365.kdocs.cn 认为本次访问**未登录**，于是做了重定向到登录。若 Cookie 已正确注入且有效，不应出现这步重定向。
- 报错 `Target page, context or browser has been closed` 是在**等待** `div:nth-child(3) > .item-title` 时出现的。该元素在测验页才有，登录页没有；且重定向/登录页上的某些行为（如 SSO、iframe、新开页）可能导致页面或上下文被关闭，从而触发该错误。所以「未登录 → 被重定向到登录页」是原因，「page closed」是后果，而不是「手速太快」。

因此：**问题本质是登录态未生效（未注入或无效），不是时序上操作太快。**

---

## 二、执行流程里 Cookie 何时生效（说明「不是操作太快」）

当前执行顺序（`backend/src/services/executor.js`）是：

1. 克隆/拉取仓库  
2. 安装依赖  
3. 把 `backend/data/run.env` 或 `backend/data/.env` 复制到仓库根 `.env`  
4. 调用 `injectAuthFromEnv(repoDir)`：用 `.env` 里的 `WPS_SID` 在仓库根生成 **auth.json**  
5. 再启动 Playwright，且通过环境变量 `AUTH_FILE` 指向该 auth.json  

Playwright 侧（`playwright.config.ts`）：

- `storageState: existsSync(authFile) ? authFile : undefined`
- 即：若 auth 文件存在，则在**每个 test 的浏览器 context 创建时**就加载该 storageState（Cookie 等），**之后第一次 `page.goto` 就会带 Cookie**。

因此：

- Cookie 的生效时机 = context 创建时，早于任何 `goto` 或点击。  
- 不存在「先点了再等 Cookie 生效」的情况；若仍被重定向到登录页，只能是：**没有加载到有效登录态**（未注入或无效）。

---

## 三、可能原因与排查

### 1. Cookie 根本没有注入（未生成 auth.json）

- **原因**：执行时没有读到 `WPS_SID`，`injectAuthFromEnv` 直接 return，不会写 auth.json。
- **配置**：  
  - 执行器读取顺序：**仓库根 .env（复制后）→ `backend/data/run.env` → `backend/data/.env` → `backend/.env`**  
  - 至少在一处配置 **`WPS_SID=你的值`**（注意键名、无空格、无引号包裹）。
- **排查**：  
  - 看**执行日志**里是否有：  
    - `[Executor] 已用仓库内 Playwright 生成 auth.json，登录态已注入` 或  
    - `[Executor] 已从 run.env/.env 手写 auth.json 注入登录态`  
  - 若两条都没有，且出现 `[Executor] 未找到 run.env 或 .env...` 或 `未在 .env/run.env 中找到 WPS_SID`，说明**没有注入**，属于「Cookie 未注入」。

### 2. Cookie 已注入但无效（过期 / 错误 / 域名不全）

- **原因**：auth.json 已生成并传给 Playwright，但服务端不认（例如 WPS_SID 过期、被踢下线、或手写 JSON 缺少某些 cookie/domain）。
- **现象**：仍然被重定向到 account.wps.cn 登录页。
- **排查**：  
  - 用本地浏览器登录 365.kdocs.cn，在开发者工具里看 Cookie（尤其是 `wps_sid`）的 domain、path、是否 HttpOnly 等。  
  - 对比 executor 手写格式（仅 `.kdocs.cn`、`.wps.cn` 的 `wps_sid`）；若实际还需要 `account.wps.cn` 等域，手写 fallback 可能不完整。  
  - 重新获取一次有效的 WPS_SID，更新 `run.env` 或 `data/.env` 后再跑一次。

### 3. 为何会报「Target page, context or browser has been closed」

- 流程是：`page.goto(测验页)` → 服务端发现未登录 → 重定向到 **account.wps.cn** → 当前页面变成登录页，**没有** `div:nth-child(3) > .item-title`。  
- 用例里接着执行 `page.locator('div:nth-child(3) > .item-title').click()`，Playwright 会一直等该元素；在等待过程中，若发生**重定向、SSO 跳转、新开/关闭窗口**等，就可能触发「page/context/browser closed」。  
- 所以这是**未登录导致页面不对、再导致等待与关闭报错**，而不是「点得太快」。

---

## 四、建议操作（按优先级）

1. **确认执行时是否真的注入了 Cookie**  
   - 看当次 run 的**后端/执行日志**是否出现上述两条「已…注入」之一。  
   - 若没有：在 `backend/data/run.env`（或 `data/.env`）中配置 `WPS_SID=...`，确保执行器能读到。

2. **确认 WPS_SID 是否有效**  
   - 浏览器登录 365.kdocs.cn，从 Cookie 里复制最新的 `wps_sid`，更新到 `run.env` 或 `data/.env`，再执行一次计划。

3. **（可选）用例里避免在错误页面上操作**  
   - 在 `page.goto('https://365.kdocs.cn/wiki/l/0sB3eZVC/qa/')` 之后：  
     - 加 `await expect(page).toHaveURL(/365\.kdocs\.cn/, { timeout: 15000 })` 或等待测验页独有元素，确认没被重定向到登录页再点 `div:nth-child(3) > .item-title`。  
   - 这样一旦未登录，会很快在「URL 或元素断言」处失败，报错更清晰，而不是误报「page closed」。

---

## 五、小结

| 问题 | 结论 |
|------|------|
| 是 Cookie 没注入，还是操作太快？ | **是 Cookie 未注入或无效**；Playwright 在 context 创建时就已加载 storageState，不存在「还没生效」的时序问题。 |
| 为何出现「page/context/browser closed」？ | 未登录 → 被重定向到登录页 → 目标元素不存在 → 等待过程中发生重定向/关闭 → 报 closed。 |
| 建议 | 先查执行日志确认是否有注入；再确认 WPS_SID 有效并更新到 run.env/data/.env；可选在 goto 后加 URL/元素断言以便区分「未登录」与其它错误。 |
